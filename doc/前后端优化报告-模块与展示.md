# 前后端优化报告：模块联系与必要展示

> 基于子代理（explore）对 Agent Team version3 的排查结果整理。**1. 模块联系** 聚焦后端模块边界与依赖；**2. 前端展示** 聚焦是否缺少必要内容与未用 API。

---

## 一、后端：模块联系与可优化点

### 1.1 当前模块链路（简要）

- **入口**：`run_backend.py` → `app`；app 挂载 `routes`（聚合各子 router）、启动时创建 `CoreAgent` + `JAgentExtension`。
- **路由 → 服务/工具**：`routes_oos`、`routes_quotation`、`routes_procurement`、`routes_upload` 直接 import 并调用 `backend.tools.*`（如 `quotation_agent_tool`、`replenishment_preview`、`execute_inventory_tool`、`email_service`），未经过统一 service 层。
- **DataService**：API 通过 `deps.get_oos_data_service()` 单例使用；`backend.agent.tools` 里 `_run_oos_*` 与 `fulfill.run_fulfill_loop` 内部仍 `DataService()` 新建实例，存在多实例与生命周期不统一。

### 1.2 已修复

- **Warmup 报错**：`app.py` 中原先调用 `api_routes._get_oos_data_service()`，`routes` 无此方法会 AttributeError。已改为从 `backend.server.api.deps` 引入 `get_oos_data_service` 并调用，与 API 共用同一单例。

### 1.3 建议优化（按优先级）

| 优先级 | 建议 | 涉及文件/区域 |
|--------|------|-------------------------------|
| **高** | **fulfill 使用 DataService 单例**：`backend/server/services/fulfill.py` 改为从 `deps` 取 `get_oos_data_service()`，不再内部 `DataService()`，与 API 一致。 | `fulfill.py` |
| **高** | **agent tools 使用 DataService 单例**：`backend/agent/tools.py` 中 `_run_oos_list`、`_run_oos_stats`、`_run_oos_by_file`、`_run_oos_by_time` 改为通过 `deps.get_oos_data_service()` 获取实例，避免每请求新建。 | `backend/agent/tools.py` |
| **中** | **薄 service 层**：在 `backend/server/services/` 下增加 OOS/报价相关薄封装（如 `oos_service.py`），对外提供 `get_stats`、`get_list`、按文件/时间查询等；routes 只调该 service，不直接 import `backend.tools.oos.*` / `backend.tools.inventory.*`，工具实现可逐步迁到 service 内或由 service 委托。 | `routes_oos.py`、`routes_quotation.py`、新建 `server/services/` |
| **低** | **路由侧工具调用收口**：将 `routes_quotation`、`routes_oos` 中对 `backend.tools.*` 的调用集中到 `server/services/` 某一模块（如 `quotation_use_cases.py`），路由只调该模块 + deps，便于后续替换实现。 | 同上 |

## 二、前端：必要内容与未用 API

### 2.1 视图与数据源（现状）

| 标签页 | 作用 | 数据来源 |
|--------|------|----------|
| Chat | 与 Agent 对话 | Gateway WS、可选文件上传 |
| Overview | 总览 | Gateway + `/api/oos/stats`、`/api/shortage/stats` |
| Channels | 业务知识 | `/api/business-knowledge`、dependent-files |
| Instances | 无货/缺货列表与操作 | `/api/oos/*`、`/api/shortage/*` |
| Sessions | 采购批准 + 补货 | `/api/shortage/list`、`/api/procurement/approve`、`/api/replenishment-drafts` |
| Work | 批量报价 | `/api/work/*`、`/api/quotation/upload`、from-text、quotation-drafts |
| Cron | 成单（Fulfill） | `/api/quotation-drafts?status=pending&limit=50`、confirm |
| Agents / Skills / Nodes / Config / Debug / Logs | 配置与运维 | Gateway |

### 2.2 缺失或未对齐的展示 / API 使用

| 序号 | 问题 | 位置 | 建议 |
|-----|------|------|------|
| 1 | **客户档位未用后端** | Work 页客户档位下拉：`control-ui/src/ui/views/work.ts` 使用硬编码 `PRICE_LEVEL_OPTIONS`；后端已有 `GET /api/config/price-levels`。 | 在 Work 视图中请求 `GET /api/config/price-levels` 作为下拉选项，与后端配置一致。 |
| 2 | **报价单列表未用服务端筛选** | Fulfill（Cron）仅请求 `status=pending&limit=50`，筛选在前端内存；后端支持 `name`、`status`、`limit`。 | 在 Fulfill 列表增加状态、名称筛选，请求时带上 `name`、`status` 等参数，由后端过滤。 |
| 3 | **无货/缺货列表缺少溯源信息** | `control-ui/src/ui/views/oos-dashboard.ts` 行信息未展示 `uploaded_at`、`file_name`；类型中已有这些字段。 | 在 OOS/Shortage 列表行中展示「首次时间」和/或「来源文件」，便于追溯。 |
| 4 | **Instances 无货/缺货无分页** | `controllers/oos.ts` 用 `limit=100`，视图用 `.slice(0, 50)` 并提示「仅展示前 50 个」；与 Fulfill、Procurement 有分页不一致。 | 为 Instances 的 OOS/Shortage 列表加分页（先客户端分页或后端支持 offset/limit/count）。 |
| 5 | **Shortage「仅未批准」未落地** | 前端 Procurement 请求 `GET /api/shortage/list` 带 `unapproved_only=1`；`routes_oos.py` 的 `shortage_list` 未实现该参数。 | 在 `GET /api/shortage/list` 实现 `unapproved_only`（或等价逻辑），或前端去掉该参数并说明列表为「全部缺货」由前端筛。 |

### 2.3 未使用或未充分使用的后端能力

- `GET /api/config/price-levels`：未接入 Work 客户档位。
- `GET /api/quotation-drafts` 的 `name` 参数：无 UI 按名称搜索。
- `GET /api/quotation-drafts/by-no/{draft_no}`：目前仅用 `id`，可按需在详情/跳转中使用 `draft_no`。

---

## 三、小结

- **后端**：已修 warmup 调错方法的问题；建议统一 DataService 单例（fulfill + agent tools），再考虑薄 service 层与路由收口，减少「路由直连 tools」的耦合。
- **前端**：建议补全「客户档位用 API」「报价单服务端筛选」「无货/缺货溯源与分页」「shortage 未批准过滤」等必要展示与参数，使前后端能力对齐、信息可追溯。
