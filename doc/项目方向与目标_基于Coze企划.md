# Agent Team version3 项目方向与目标（基于 Coze 企划）

> 本文档由《Coze 助力 PT. VANTSING 企业微信 + ACCURATE 系统全业务流程实施方案》转化而来，明确 **本项目要努力达成的方向** 与 **优先落地的业务闭环**，供迭代开发与排期参考。
---

## 一、总方向（项目要努力做到的事）

1. **全流程在单一入口完成**  
   用户在企业微信（或当前控制台/聊天）内完成：报价、库存查询、成单、采购触发、送货跟踪、收款提醒与记录，无需切换多系统。

2. **数据不臆造、以系统为准**  
   涉及价格、库存、订单、采购、物流、收款等具体数据时，**必须**通过工具调用 ACCURATE/本地数据源或万鼎价格库获取，严禁 Agent 自行编造；数据写入后需与后端/ACCURATE 保持一致。

3. **打通数据壁垒，减少人工重复**  
   报价、成单、无货/缺货、采购、收款等环节产生的数据，应落库并可被后续流程复用（如报价→成单→库存锁定→采购→收款），避免人工重复录入、重复查询。

4. **合规与安全**  
   遵循数据与隐私合规要求，不采集、不泄露客户敏感信息；与外部系统（ACCURATE、企微）的对接需权限可控、传输加密、异常可追溯。

5. **意图识别与话术标准化**  
   支持自然语言 + 标准化指令（如「报价+型号+数量」「库存查询+型号」「成单+客户+报价单号」），提升识别准确率，便于与企微/ACCURATE 流程对齐。

## 二、六大业务闭环（要努力实现的方向）

以下六条对应企划中的 6 大环节，每条的「要努力做到」即本项目应努力实现的方向。

### 2.1 报价闭环

- **方向**：客户/销售发起询价（自然语言或「报价+管道型号+数量+交货地址」）→ 系统调取产品单价、折扣、库存（ACCURATE/万鼎）→ 自动计算并生成报价单 → 推送结果并落库，便于后续成单与统计。
- **version3 现状**：已有万鼎选型、历史匹配、报价填表（Work Mode）、库存查询；Chat 内可询价、可查库存。
- **要努力做到**：  
  - 报价结果可落库（报价台账），并与后续「成单」环节打通（成单时能带出报价记录）。  
  - 支持「同意报价」等简短确认后，一键进入成单流程（报价→成单无缝衔接）。  
  - 报价数据与 ACCURATE 报价/订单模块同步（若已对接 ACCURATE）。

### 2.2 库存查询闭环

- **方向**：客户/销售发送「库存查询+型号」或自然语言 → 系统从 ACCURATE/本地数据源拉取实时库存（数量、可用库存、安全库存）→ 反馈结果并记录查询日志；低于安全库存时触发缺货/采购预警。
- **version3 现状**：已有 inventory 查询（含按 code、按关键词）、无货/缺货登记与邮件提醒。
- **要努力做到**：  
  - 库存数据来源统一为 ACCURATE 或与 ACCURATE 同步的本地库，避免多源不一致。  
  - 缺货时能推荐替代型号（同规格、可兼容）并带出库存与报价。  
  - 查询记录落库，支撑「查询频次分析、库存布局优化」。

### 2.3 成单闭环

- **方向**：客户确认报价或销售录入订单 → 生成标准订单（订单号、客户、产品、总价、交期、付款方式、地址）→ 订单写入 ACCURATE/本地 → 锁定库存，避免超卖；并通知库存/采购/物流群。
- **version3 现状**：Work Mode 支持报价表填充与缺货统计；Chat 可询价、查库存，暂无「成单」专用工具与 ACCURATE 订单写入。
- **要努力做到**：  
  - 成单流程工具化：从报价记录或表单生成订单，写入后端/ACCURATE，并返回订单号与状态。  
  - 成单后自动触发库存占用/锁定逻辑（或推送至库存/采购侧）。  
  - 支持「手动成单+详细订单信息」的解析与落库。

### 2.4 采购闭环

- **方向**：库存低于安全库存或成单后库存不足时，触发采购需求 → 计算采购量（安全库存 − 当前库存 + 订单需求）→ 生成采购单（供应商、数量、交期）→ 采购单写入 ACCURATE → 跟进采购进度（待下单/已下单/待到货）→ 到货后更新库存并同步订单状态。
- **version3 现状**：无独立采购模块；无货/缺货可触发邮件，未与采购单、供应商、入库联动。
- **要努力做到**：  
  - 在无货/缺货或成单锁库不足时，能生成「采购需求」或采购单草稿，并写入 ACCURATE（若已对接）。  
  - 采购进度可录入并在系统内查询；到货后能更新库存与订单状态（采购-入库闭环）。

### 2.5 送货/物流闭环

- **方向**：备货完成后生成送货单 → 物流安排与单号绑定 → 物流状态实时可查并同步至客户与内部 → 客户确认收货后更新订单状态为「已收货」，并写入 ACCURATE。
- **version3 现状**：无送货单、物流跟踪、收货确认等专用能力。
- **要努力做到**：  
  - 支持「备货完成+订单号」触发送货单生成与物流任务。  
  - 物流状态可录入或对接物流 API，并在聊天/看板中展示；支持「已收货」确认并回写订单状态。

### 2.6 收款闭环

- **方向**：按订单付款条款（如预付款 30%、发货前付清）定时提醒客户付款 → 客户付款后（凭证或企微支付）确认到账 → 收款信息写入 ACCURATE 财务模块，订单状态更新为「已收款」；逾期未付则催收提醒（规则化频次与话术，避免骚扰）。
- **version3 现状**：无收款提醒、到账确认、与 ACCURATE 财务同步的能力。
- **要努力做到**：  
  - 根据订单付款条款自动触发「收款提醒」（时间、金额、方式）。  
  - 支持「已付款/已到账+订单号」的确认与落库，并更新订单状态。  
  - 收款数据与 ACCURATE 财务模块同步（若已对接）；催收逻辑合规、可配置。

## 三、数据与系统对接（要努力达成的能力）

| 方向 | 说明 |
|------|------|
| **企微作为主入口** | 当前以控制台/聊天为主；若部署到企微，需支持企微 API（消息接收、回复、群@、客服/机器人），话术与企划中的指令格式对齐。 |
| **ACCURATE 为数据核心** | 库存、订单、采购、财务等以 ACCURATE 为准（或与 ACCURATE 同步的本地库）；Agent 只读/写通过统一 API，不臆造数据。 |
| **数据同步规则** | 库存可定时或按需同步；订单/成单/收款建议实时或近实时同步；同步失败有日志与告警（如 @管理员），便于排查。 |
| **权限与安全** | 对接 ACCURATE/企微的账号权限分级；传输加密；敏感数据不落日志、不泄露。 |

## 五、建议实施优先级（分阶段努力）

- **P0（优先）**  
  - 报价→成单数据打通（报价可落库、成单可带出报价）。  
  - 库存来源统一与缺货时的替代型号推荐。  
  - 无货/缺货与采购需求或采购单的衔接（至少生成采购需求/草稿）。

- **P1（其次）**  
  - 成单工具与订单写入 ACCURATE/本地。  
  - 收款提醒与到账确认、收款数据落库并同步 ACCURATE（若已对接）。  
  - 企微入口对接（若业务需要）。

- **P2（后续）**  
  - 采购单全流程（生成→跟进→入库→更新库存与订单）。  
  - 送货单与物流状态、收货确认与订单状态回写。  
  - 数据同步监控、异常告警与合规审计。

## 六、Work 模式改造思路（报价单→落库→确认→下游闭环）

在保持现有 ReAct + workflow、由 LLM 判断是否进入下一步的前提下，对 Work 模式做如下改造，使**待确认报价单**成为连接报价与成单/采购/送货的枢纽。

### 6.1 整体流程

1. **输入**  
   - **报价单**（上传 Excel）或 **自然语言**（如「报 50 三通 100 米、dn40 弯头 50 个」）。  

2. **第一步：查价格**  
   - 系统对每条询价项做价格匹配（历史 + 万鼎等），得到产品标号、单价、档位等。  

3. **第二步：查库存**  
   - 按产品标号/规格查库存，得到可用库存、安全库存、是否缺货等。  

4. **生成待确认报价单**  
   - 汇总为一份**初步的、待确认的报价单**，字段包含但不限于：产品标号、产品名称、规格、需求数量、单价、金额、库存（可用/安全）、缺货标记、匹配来源等。  

5. **落库并给编号**  
   - 将上述报价单数据**落库到云端数据库**（与现有无货/缺货等库分离或同库分表），并生成**唯一编号**（如报价单号 / 草稿 ID），便于后续引用与追溯。  

6. **报价员检查 → 用户确认**  
   - 报价员在系统内检查、修正待确认报价单（可选）；  
   - 用户（或销售）确认报价单后，该单状态更新为「已确认」，方可进入下游。  

7. **进入下游闭环**  
   - **已确认**的报价单，允许进入：  
     - **2.3 成单闭环**：转成正式订单、写 ACCURATE/本地、锁库存；  
     - **2.4 采购闭环**：库存不足时生成采购需求/采购单；  
     - **2.5 送货/物流闭环**：备货完成后的送货单、物流跟踪、收货确认；  
     - 以及后续 **2.6 收款闭环** 等。  

### 6.2 要点小结

- **报价单 or 自然语言** 均可作为入口，统一产出「待确认报价单」并落库、给编号。  
- **先价格、后库存** 的顺序固定为 workflow 步骤，中间结果与最终待确认报价单均落库，便于审计与进入成单/采购/送货。  
- **编号** 作为后续成单、采购、送货时引用「来自哪张报价单」的关联键。  
- **报价员检查 + 用户确认** 作为门控，确认后才允许进入成单等下游环节，与「由 LLM 判断是否可以下一步」并行使用（如：LLM 判断「用户已确认报价单」后再触发成单 workflow）。
