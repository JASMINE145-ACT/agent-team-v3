# Agent Team v3 代码准则

> 基于对本项目实际架构审查（2026-03）提炼，针对本项目真实问题，不是通用教条。

---

## 一、依赖与分层

### 规则 1：依赖方向只能向下，绝不向上
```
core → (无业务依赖)
plugins/jagent → core, tools
tools → (无 server 依赖)
server/api → plugins, tools, core
```

**禁止**：
- `tools/` 里 import `server/api/` 的任何模块
- `core/` 里 import `backend.config.Config`（应通过构造函数参数注入）

**正确做法**：
```python
# ❌ 错误：core 层直连配置
from backend.config import Config
class CoreAgent:
    def __init__(self):
        self.max_tokens = Config.MAX_TOKENS  # 耦合

# ✅ 正确：配置通过参数注入
class CoreAgent:
    def __init__(self, max_tokens: int = 4096, ...):
        self.max_tokens = max_tokens
```

---

### 规则 2：跨层访问必须通过单一入口

每一层只能依赖紧邻的下一层，不能跨层。
访问 DataService 统一走单例入口，不在各处直接 `DataService()`。

**说明**：单例当前在 `server/api/deps.py`。若在 **server 或 app 启动链路上的代码**（如 routes、fulfill、warmup），直接 `from backend.server.api.deps import get_oos_data_service` 使用。若在 **tools/agent** 中需要 DataService，优先通过**参数注入**（由 Extension 或调用方从 deps 取 ds 后传入），避免 tools 层 import server，与规则 1 一致；若暂时仍从 deps 取，则视为例外并加注释，后续迁为注入或把单例挪到 `backend/core` 等非 server 层。

```python
# ❌ 错误（每次调用新建实例）
from backend.tools.oos.services.data_service import DataService
ds = DataService()

# ✅ 正确（server 侧：走单例入口）
from backend.server.api.deps import get_oos_data_service
ds = get_oos_data_service()
```

## 二、全局状态管理

### 规则 3：module-level 全局变量只能有一种写法

允许的唯一模式是**延迟初始化单例**，必须配 lock（多线程场景）或限制在只有一个调用方的情况下使用：

```python
# ✅ 允许的单例模式
_service: Optional[DataService] = None

def get_service() -> DataService:
    global _service
    if _service is None:
        _service = DataService()
    return _service
```

**禁止**：
- 散落在各模块里的 `_xxx = None` 不成体系地复制这个模式（当前项目有 12+ 处）
- 新增单例前，先问：能不能放进 `CoreAgent` 或 `Extension` 的实例属性？

**目标**：新功能的共享资源统一挂在 `app.state` 下，不新增 module-level 全局变量。

## 三、单文件职责

### 规则 4：一个文件不超过 200 行有效逻辑（不含注释/空行）

超过时拆分，按**领域**拆而不是按「大小」拆：

| 当前问题 | 拆法 |
|----------|------|
| `plugins/jagent/extension.py` 240+ 行，管理 9 个工具 | 拆为 `tools/oos/handler.py`、`tools/inventory/handler.py`、`tools/quotation/handler.py`，extension 只做聚合 |
| `backend/agent/tools.py` 承担工具定义+实现+schema | 工具 schema 放 `EXTRA_TOOLS`（已有），实现放各 `_run_*` 函数——当前已合理，后续新工具不要继续加在这里 |

---

### 规则 5：handler 不要重复写，提取公共包装器

当前有多处相同的 handler 返回模式：
```python
out.get("result", "") if out.get("success") else json.dumps(out, ensure_ascii=False)
```

提取一次：
```python
# backend/core/tool_utils.py（已有，扩展即可）
def unwrap_tool_result(out: dict) -> str:
    if out.get("success"):
        return out.get("result", "")
    return json.dumps(out, ensure_ascii=False)
```

**原则**：同一段逻辑出现 2 次就提取，出现 3 次一定是 bug 温床。

## 四、错误处理

### 规则 6：禁止无声吞异常

```python
# ❌ 绝对禁止
try:
    ...
except Exception:
    pass

# ❌ 不够，debug 级不会在生产中可见
except Exception:
    logger.debug("failed")

# ✅ 最低要求
except Exception:
    logger.warning("xxx 失败，已跳过", exc_info=True)

# ✅ 关键路径必须用 exception 级
except Exception:
    logger.exception("xxx 失败")
    raise  # 或返回错误结果
```

**例外**：扩展（Extension）的 `on_before_prompt` 可以吞掉并 warning，但必须记日志。

---

### 规则 7：API 层统一返回格式

所有 routes 返回相同结构，前端只需检查一个字段：

```python
# ✅ 统一格式
{"success": True,  "data": {...}}
{"success": False, "error": "具体原因"}

# ❌ 禁止混用
{"status": "error", "message": ...}   # routes_chat 当前写法
{"answer": ..., "trace": ...}          # 直接裸返回
```

新增路由必须使用 `success/data/error` 三字段，不得引入新的顶层字段名。已有路由（如 routes_chat 的 `status`/`message`）可逐步迁移，新代码不再引入其他格式。

## 五、类型注解

### 规则 8：公共函数必须有完整类型注解

```python
# ❌ 不合格
def _make_oos_handler(self, fn, arg_name, default):
    async def handler(args, context):
        ...

# ✅ 合格
from typing import Any, Callable, Awaitable

def _make_oos_handler(
    self,
    fn: Callable[..., dict],
    arg_name: str,
    default: Any,
) -> Callable[[dict, dict], Awaitable[str]]:
    async def handler(args: dict, context: dict) -> str:
        ...
```

**边界**：私有内部函数（单下划线，仅在同文件用）可以省略，但公共接口（被其他文件 import 的）一律加。

## 六、Import 规范

### 规则 9：在函数体内 import 只允许两种情况

1. **延迟加载**：模块初始化慢（如加载大 Excel），在第一次调用时才 import
2. **打破循环依赖**：两个模块互相依赖时的最后手段

其他情况统一写在文件顶部。

```python
# ❌ 错误：普通逻辑写在函数体内的 import
def _run_oos_list(limit: int = 100) -> dict:
    from backend.server.api.deps import get_oos_data_service  # 应放顶部
    ds = get_oos_data_service()

# ✅ 正确：顶部 import（本项目 tools.py 改完后应该这样）
from backend.server.api.deps import get_oos_data_service

def _run_oos_list(limit: int = 100) -> dict:
    ds = get_oos_data_service()
```

## 七、前端准则（control-ui）

### 规则 10：Controller 负责数据，View 负责渲染，二者不混

```typescript
// ❌ 错误：在 View 里直接发 fetch
function renderOosList() {
    fetch("/api/oos/list").then(r => r.json()).then(data => { ... })
}

// ✅ 正确：View 只接受 props，Controller 负责 fetch
// controller/oos.ts
async function loadOos(state, limit = 100) {
    const data = await apiFetch("/api/oos/list", { limit });
    state.oosList = data;
}

// views/oos-dashboard.ts
function renderOosList(props: { list: OosItem[], onDelete: Fn }) {
    return html`...`;
}
```

### 规则 11：硬编码选项必须先查后端是否已有 API

当前反例：`work.ts` 里 14 个价格档位硬编码，但后端已有 `GET /api/config/price-levels`。

**操作流程**：
1. 写前端下拉/选项前，先搜 `routes_*.py` 是否已有对应 API
2. 有则调 API；无则与后端约定接口后再实现，禁止在前端长期维护与后端业务重复的常量表

## 八、新功能开发 Checklist

每次新增功能时对照检查：

- [ ] 依赖方向正确（不向上依赖）
- [ ] 没有新增 module-level 全局变量（或有充分理由并加注释）
- [ ] 错误处理：关键路径用 `logger.exception()`，不吞异常
- [ ] API 返回格式用 `{"success": bool, "data"/"error": ...}`
- [ ] 公共函数有类型注解
- [ ] 没有将同一逻辑写两遍（先搜再写）
- [ ] 前端选项/配置先确认后端是否有 API

---

## 九、不要做的事（Anti-patterns 速查）

| 反模式 | 替代方案 |
|--------|----------|
| `DataService()` 在工具函数里新建 | `get_oos_data_service()` 取单例 |
| `except Exception: pass` | 至少 `logger.warning(..., exc_info=True)` |
| 路由返回 `{"status": "error", "message": ...}` | `{"success": False, "error": ...}` |
| 前端写死下拉常量 | 调 `/api/config/xxx` 动态加载 |
| 一个文件里注册所有工具（240+ 行） | 按领域拆为独立 handler 文件 |
| `from module import X` 写在函数体内（非必要） | 移到文件顶部 |
| 同一返回格式逻辑写 3 次 | 提取到 `tool_utils.py` |
