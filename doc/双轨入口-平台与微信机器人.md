# 双轨入口：平台软件 + 微信机器人

> 目标：**两条使用方式**——（1）**平台软件**：当前控制台，高度人工操作；（2）**微信机器人**：像业务员一样在群里交互，**产生的数据统一进入平台**，在平台里做审核、成单、采购批准等。Boss 诉求：支持微信机器人作为一线入口，数据回流平台。

---

## 1. 双轨定位

| 入口         | 角色               | 典型使用场景 |
|--------------|--------------------|--------------|
| **平台软件** | 人工操作中心       | 报价页上传 Excel、跑 Work、确认并保存；成单页勾选确认；采购页批准缺货；补货确认；查看无货看板、业务知识、配置与日志。 |
| **微信机器人** | 一线「业务员」入口 | 在群里回复「报 50 三通 100 米」→ 生成报价草稿并落库；「查库存 dn40 弯头」→ 返回结果；「这批缺货帮我登记」→ 落缺货/触发邮件。群里产生的**待决数据**在**平台**里展示，由人在平台（或 CEO 模式）批准/成单/补货。 |

**核心**：微信机器人**不替代**平台，而是**多一个入口**；数据写同一套后端与库，平台仍是「处理、审核、决策」的中心。

---

## 2. 数据流（微信 → 平台）

```
群里用户发消息
    ↓
微信机器人（接收、解析意图）
    ↓
调用 version3 现有 API（与平台共用）
    ↓
数据落库（quotation-drafts、shortage、replenishment-drafts、oos 等）
    ↓
平台软件里可见：报价列表、缺货列表、补货列表、无货看板等
    ↓
人在平台做「确认并保存 / 成单 / 批准 / 补货确认」或开 CEO 模式自动批
```

**可复用的 API 示例**（无需为微信单独造一套）：

- 报价：`POST /api/quotation/from-text`（自然语言 → 报价草稿）、`POST /api/quotation-drafts`（若机器人先拼好 lines）
- Work 跑报价：`POST /api/work/run`（若需走完整查价+库存流程，可带 context）
- 库存查询：`POST /api/query` 或现有 chat/query（inventory 能力）
- 无货/缺货：现有 oos、shortage 相关 API，机器人可调接口落库并回复「已登记，平台可查」
- 补货：`POST /api/replenishment-drafts`（机器人解析「补 XX 多少件」→ 调预览+落库）

**不在微信里做的**：复杂表格编辑、批量勾选、审批决策（成单/采购批准/补货确认）——这些留在**平台**做，或由平台侧开 CEO 模式由 LLM 批。

---

## 3. 实现要点（尽量少造轮子）

1. **微信侧**  
   - 接微信/企微消息 API（接收群消息、@机器人、发回复）。  
   - 意图识别：自然语言 → 对应「报价 / 查库存 / 登记无货 / 补货 / 其他」；可复用现有 `query`、`from-text`、Work 的解析与调用。  
   - 收到消息后调 version3 后端 **HTTP API**（与平台同一 host 或内网可见），写入同一 DB。

2. **后端**  
   - **不新增业务逻辑**：报价、缺货、补货、成单、采购批准仍用现有 API；如需「来源」区分，可在落库时加 `source=wechat` 或 `source=platform`。  
   - 可选：为微信增加一个 **webhook 或最小路由**（如 `POST /api/wechat/callback`），只做验签、解包消息、转成内部调用（调现有 /query、/quotation/from-text、/work/run 等），回复内容再通过微信 API 发回群里。

3. **平台**  
   - 列表/看板支持按 `source` 筛选（可选），便于区分「来自微信」与「来自平台」的待办；业务逻辑（成单、批准、补货确认）不变，仍可接 CEO 模式。

---

## 4. 小结

| 项目     | 做法 |
|----------|------|
| 平台软件 | 保持现状：高度人工，报价/成单/采购/补货在控制台完成 |
| 微信机器人 | 群里交互，像业务员：询价、查库存、登记无货/缺货、发起补货草稿；**数据全部进同一后端与 DB** |
| 数据流   | 微信 → 现有 API → 落库 → 平台展示与处理（人工或 CEO） |
| 实现     | 微信接消息 + 意图识别 + 调现有 API；后端尽量只加 wechat callback 或最小路由，不重复造业务 |

这样 Boss 要的「微信机器人」和「平台软件」两条线都能满足：机器人在群里产生数据，平台继续做重操作与审批，数据统一、不双轨分家。
