## 二、架构层面的问题

### 1. System Prompt 体积过大（效率核心问题）

```
System Prompt 估计 5000-8000 tokens
├─ 固定前缀: ~50 tokens
├─ SKILL_INVENTORY_PRICE: ~2000 tokens
├─ SKILL_EXCEL_CHAT: ~800 tokens
├─ SKILL_CLARIFY: ~300 tokens
├─ SKILL_KNOWLEDGE: ~300 tokens
├─ OUTPUT_FORMAT: ~1500 tokens
└─ GLOBAL_HARD_CONSTRAINTS: ~150 tokens
```

**每次 LLM 调用都要带 6000+ tokens 的固定开销**，即使查询是"你好"这样的简单输入。这直接推高每次调用的成本和延迟。

### 2. 多层注入叠加，信息冗余严重

用户说"查一下"，实际注入到 `messages[role=user]` 的内容可能包含：

```
[会话上下文 — 最近 4 轮]          ← build_injection()
[最近工具调用]                    ← build_tool_memory_injection()
【当前主题】上一轮问: xxx，用户本句: yyy  ← U型注意力绑定
```

**四类 injection 说的是同一件事**：当前要做什么。堆在一起让 system prompt 膨胀，且给模型造成干扰。

### 3. 两套 Agent 路径维护成本高（已完成）

状态：`DONE`（2026-03-30）

```
agent_runner.py
└─ run_inventory_agent() 已改为内部调用 SingleAgent.execute_react()
   （复用 core/agent.py + skills.py 路径，不再维护独立 _system_prompt）

core/agent.py
└─ build_system_prompt(skill_parts, output_fmt) 继续作为统一 Prompt 组装入口
```

本项已落地：inventory 路径不再维护独立内联 prompt，`skills.py` 更新只需维护一条主路径，避免双路径漂移。

### 4. skills.py 的 DOC/RULES 双版本 = 两倍维护负担

| 常量 | DOC 版本 | RULES 版本 | 功能 |
|------|---------|-----------|------|
| `SKILL_INVENTORY_PRICE` | 2000 tokens | 2000 tokens | 同一份逻辑，两种描述风格 |
| `SKILL_EXCEL_CHAT` | 800 tokens | 800 tokens | 同上 |
| `OUTPUT_FORMAT` | ~100行 | ~100行 | 同上 |

每次修改规则需要同时改两个版本，生产验证通过后才能确认改动正确，**迭代成本翻倍**。

## 三、Prompt 设计层面的问题

### 5. 中文关键词保护规则不可靠

```
keywords 保护规则：
中文管件名称词——「直接（接头）」「直通」「弯头」「三通」...
——必须原样保留在 keywords 中，禁止去除。
```

这条规则在 `skills.py` 里是**纯自然语言约束**，靠模型自觉遵守。没有任何代码层面的强制手段。从 journal bug 记录看，这个问题已经在生产中引发过实际的选错工具 bug。

### 6. 档位映射散落在 Prompt 里，没有结构化数据

```
用户说「二级代理」→ A
用户说「一级代理」→ B
用户说「聚万大客户」→ C
用户说「青山大客户」→ D
```

这个映射在 Prompt 里是自然语言描述，**不是代码**。如果要新增档位或改映射，必须改 Prompt、重启、验证，没有结构化的配置表（如 JSON/YAML）来管理这部分。

### 7. Rework 触发词是硬编码列表

```python
_REWORK_KEYWORDS = ["错了", "不对", "不是这个", "重新选", "换一个", ...]
```

这个列表如果想扩展（如增加「选错了」「重新选一个」），需要改 Python 代码、验证、重启。**不够灵活**。

## 四、效率层面的问题

### 8. 上下文截断策略不透明

```python
def _trim_context(messages, max_chars, id_to_name, summarizer):
    # 策略：对 tool messages 进行压缩/摘要
    # summarizer 是独立 LLM（gpt-4o-mini），用于生成 tool 结果摘要
```

当前代码实况（以 `backend/core/context_compression.py::_trim_context` 为准）：

- 触发条件：`messages` 总字符数超过阈值才触发压缩。  
  - Chat 路径阈值：`CONTEXT_MAX_CHARS`（`backend/core/agent.py`，默认 `16000`）。
  - Work ReAct 路径阈值：`_WORK_CONTEXT_MAX_CHARS`（`backend/agent/work_executor.py`，默认 `8000`）。
- 压缩对象：仅 `role=="tool"` 且 `content` 长度 `>200` 的消息。
- 压缩顺序：从最旧 tool message 开始逐条压缩，直到总长度回到阈值内。
- 非压缩对象：`system` / `user` / `assistant` 在该阶段不会被压缩。
- 摘要器：优先 LLM 摘要（默认 `gpt-4o-mini`），失败回退规则摘要；每条摘要目标上限 `800` 字。

**结论**：当前实现没有“保留最新 2 条 tool messages”的硬编码规则；它是“从旧到新逐条压，压到够为止”的策略。

### 9. Tool Memory 注入条件设计有缺陷

```python
if len(user_input.strip()) <= 20 and not (ctx.get("file_path") or ctx.get("file_id")):
    tool_injection = build_tool_memory_injection(session, max_items=3)
```

当前代码实况（`backend/core/agent.py::CoreAgent.execute_react`）：

- 仅在以下条件同时满足时注入 Tool Memory：
  - 会话存在历史 turn；
  - 当前输入长度 `<=20`；
  - 本轮没有 `file_path/file_id` 上下文；
  - 注入条数上限 `max_items=3`。

风险点：

- **短输入优先注入历史工具记忆**，但短输入通常语义不完整（如“价格”“库存”“查一下”），更容易被历史结果带偏。
- 触发条件只看输入长度，不看当前任务类型/意图匹配，容易把“长度特征”误当成“上下文相关性”。

补充：在 `_trim_context` 阶段只压缩 tool messages；历史对话本身（system/user/assistant）不在该阶段压缩，主要受注入轮数与注入构建逻辑影响。

### 10. JSON Schema 校验静默跳过

```python
if jsonschema is None:
    return args, None  # 跳过校验（向后兼容）
```

生产环境如果没有安装 `jsonschema`，参数校验完全失效。这个 fallback 在文档里没有说明，**危险且隐蔽**。

## 五、问题优先级排序

| 优先级 | 问题 | 影响 |
|--------|------|------|
| P0 | System Prompt ~6000 tokens 固定开销 | **每次调用额外成本和延迟** |
| P0 | 两套 Agent 路径分裂风险 | **Prompt 与代码行为不一致** |
| P1 | JSON Schema 校验静默跳过 | **危险：无效参数直达 handler** |
| P1 | 中文关键词保护无强制手段 | **journal 已记录的真实 bug** |
| P1 | 档位映射散落在 Prompt 自然语言里 | **维护成本高，易出错** |
| P2 | DOC/RULES 双版本维护翻倍 | **迭代效率** |
| P2 | Tool Memory 注入逻辑与需求反向 | **短输入场景下干扰模型** |
| P2 | _trim_context 压缩策略不透明 | **不可预测的截断结果** |
| P3 | 章节编号重复 / 孤余片段 | **文档质量** |
| P3 | Rework 触发词硬编码 | **灵活性** |

## 六、根本性建议

**如果只做一件事**：将 `skills.py` 的 prompt 内容**结构化**（JSON/YAML 配置表），去掉 DOC/RULES 双版本，改为通过配置字段生成 prompt。这能同时解决体积，维护性、档位映射、触发词灵活性等多个问题，同时为未来 hot-reload 铺路。
