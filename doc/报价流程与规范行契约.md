# 报价流程：当前实际执行与规范行契约

## 一、当前实际执行流程（与成熟方案对应）

| 成熟方案步骤 | 当前实现 |
|-------------|----------|
| PDF/文本 → AI 提取 JSON | 文字：`text_to_inquiry_items`（LLM + 规则兜底）→ `[{ product_name, specification, qty }]`；Excel：`extract_inquiry_items`（表头/规则，无 LLM） |
| 程序模板引擎 | `fill_template_with_inquiry_items`（只填询价列）+ `fill_quotation`（价格/规格回填） |
| 填 Excel 模板 | 同一套凌威模板：先填 B/C/E（询价），再填 G/H/J/L/N/O（报价） |

**具体步骤：**

1. **文字 → 询价行（JSON）**  
   入口：`POST /api/quotation/from-text` → `routes_upload.quotation_from_text`；解析：`text_to_inquiry.text_to_inquiry_items(text)` → `[{ product_name, specification, qty }, ...]`。LLM 有 key 时用 Chat Completions 出 JSON；失败则规则兜底。

2. **询价单模板首次填充（只写名称/规格/数量）**  
   `quote_tools.fill_template_with_inquiry_items(template_path, items, output_path, sheet_name="询价单")`；从第 3 行起 A=序号、B=product_name、C=specification、E=qty。

3. **Work：extract → match → fill**  
   extract：`work_tools._run_work_quotation_extract` → `quote_tools.extract_inquiry_items`（表头+规则，无 LLM）。  
   match：`_run_work_quotation_match` → `match_and_inventory.match_price_and_get_inventory`，产出 `fill_items_merged`（每项含 row, code, quote_name, unit_price, qty, specification）及可选 `needs_human_choice` / `pending_choices`。  
   fill：`_run_work_quotation_fill` → `quote_tools.fill_quotation`，按列 G/H/J/L/N/O 写产品编号、报价名称、报价产品规、数量、单价、总价及合计/日期。

4. **待确认报价草稿（pending_quotation_draft）**  
   在 `work_tools` 中：match 成功且非 `needs_human_choice` 时、以及 `merge_work_pending_choices` 后，由**规范行**单点构建（见下），产出 `pending_quotation_draft.lines` 与 `fill_items_for_excel`。

5. **最终 Excel 输出**  
   即 fill 结果：`fill_quotation` 写 `*_filled.xlsx`；下载：`GET /api/quotation/download?path=<basename>` 从 UPLOAD_DIR 取文件。

## 二、规范行 schema 与 fill_items 契约

**规范行**（canonical line）在 match 之后**只在一处**由 `backend/tools/quotation/canonical_lines.build_canonical_quotation_lines` 构建，规格（询价规格 + 报价产品规）**仅在此模块内**由规则 + 可选 `extract_specs_batch_llm` 产出，其它模块不再改写 `specification` / `quote_spec`。

- **规范行字段**（与 `pending_quotation_draft.lines` 每项一致）：  
  `row`, `row_index`, `product_name`, `specification`, `qty`, `code`, `quote_name`, `quote_spec`, `unit_price`, `amount`, `available_qty`, `shortfall`, `is_shortage`, `match_source`。  
  - `specification` = 询价规格（来自 item/fi）。  
  - `quote_spec` = 报价产品规（规则 `extract_spec_from_quote_name` + 可选批量 LLM）。

- **fill_quotation 入参契约**：  
  推荐使用由规范行导出的 fill_items（`canonical_lines.fill_items_from_canonical_lines`）。fill_items 每项需含：`row`, `code`, `quote_name`, `unit_price`, `qty`, `specification`。写入 Excel 列 J（报价产品规）时使用传入的 `specification`，即应由调用方传入**规范行中的 quote_spec 或 specification**，以保证表格与草稿一致。  
  管道侧（work_executor）在调用 fill 时优先使用 match/merge 产出的 `fill_items_for_excel`（即由规范行导出），若无则回退为 `fill_items_merged`。

## 三、规格单源说明

规格（询价规格、报价产品规）**只在一个地方**产出：`build_canonical_quotation_lines` 内。该函数接收 `fill_items_merged` + `items` + `shortage`，按行对齐并计算 amount/shortfall/available_qty 等，然后对每行设置 `specification`（来自 fi/item）与 `quote_spec`（规则 + 可选 `extract_specs_batch_llm`）。返回的规范行列表同时作为：

- `pending_quotation_draft.lines`（前端可编辑表格与落库）；
- `fill_items_for_excel` 的来源（经 `fill_items_from_canonical_lines` 导出），供 `fill_quotation` 写 Excel。

因此表格中列 J、草稿中的「报价产品规」、以及 API 中传的规格均来自同一规范行，避免多处解析导致不一致。
