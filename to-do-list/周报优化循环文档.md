- 主流程为两阶段：
  - Phase 1 同步：抓取 Accurate 发票 -> pandas 分析 -> 生成 `report_md` -> 入库。

### 2) 前端现状

- 周报页在 `control-ui/src/ui/views/reports-tab.ts`，已具备：

## 二、周报“内容逻辑机制”可行优化措施

## A. 内容层（报告可读性、决策价值）

- 增加固定结构：`一页总览 -> 异常点 -> 归因 -> 行动建议 -> 下周预测区间`，减少“有数据但难决策”。
- 增加“异常规则卡”：
  - 单日销售额突降/突增阈值；
  - 订单数与销售额背离（客单价异常）；
  - Top 客户集中度过高风险（CR3/CR5）。
- 将 `analysis_md` 分段标准化输出（固定标题），便于前端渲染锚点导航和后续自动摘要。

## B. 逻辑层（数据正确性、可解释性）

- 在 Phase 1 入库前新增数据质量检查（DQ Gate）：
  - 周区间完整性；
  - `total_order_count` 与 `daily_stats` 聚合一致性；
  - 金额非负与异常值检测（可配置阈值）。
- 增加“环比基线兜底”：
  - 若上周缺失，回退最近可用周；
  - 同时在报告中明确“基线来源周”，避免误解。
- 把 `llm_analyzer` 的预测部分改为“公式化输入”：
  - 先在后端算出趋势指标（均值、方差、周内波动、Top 客户稳定性）；
  - LLM 仅解释，不让其自行隐式推导关键数字。

## C. 机制层（稳定性、可运维性）

- 将 Phase 2 从进程内线程升级为可观测后台任务（队列/worker）：
  - 避免进程重启导致分析中断；
  - 提供重试次数、死信、耗时统计。
- 增加 SLA 指标：
  - 抓取耗时、分析耗时、失败率、超时率；
  - 暴露到管理面板。
- 调度增强：
  - 增加“错过调度自动补跑”（例如服务重启后补上上一周期）。

## 三、前端可行优化措施

## A. 交互与信息架构

- 历史记录列表支持筛选/搜索（按日期、状态、触发方式），避免记录变多后定位困难。
- 详情页新增“摘要卡片”：
  - 环比变化、异常告警数量、Top 客户集中度；
  - 默认先看结论，再看明细图表。
- 分析内容增加目录锚点（趋势/环比/建议/预测），提高可读性。

## B. 性能与实时性

- 将轮询升级为“优先 SSE/WebSocket，失败回退轮询”：
  - 减少无效请求；
  - 提升状态变更感知实时性。
- 图表数据量增大时做降采样与懒渲染（先渲染总览，再按需渲染明细）。
- 历史记录超过阈值后使用分页或虚拟列表，避免长列表渲染压力。

## C. 可用性与运维友好

- 超时文案附带“可执行下一步”（检查 token、检查后端日志、点击重试）。
- 在 UI 展示分析任务时长与最后错误摘要，减少排障成本。
- 任务配置区增加 cron 预览（下一次执行时间）与时区提示，降低误配置概率。

## Issue P0-1：DQ Gate 结果落库与展示

- 目标：
  - 在报告生成后输出 DQ 检查结果到 `summary_json` 或 `dq_checks_json`。

- 风险：
  - 规则过严导致误报，影响业务信任。
- 回滚方案：
  - DQ 改为仅告警不阻断；通过 feature flag 关闭展示。

## Issue P0-2：reanalyze 可观测性增强

- 目标：
  - `POST /reanalyze` 返回统一可观测字段（`record_id`, `status`, `requested_at`）。
- 验收标准：
  - API 文档明确字段语义；
  - 前端成功提示包含状态与时间戳；
  - 错误返回保留 409/422 语义区分。
- 风险：
  - 前端旧逻辑依赖当前返回结构。
- 回滚方案：
  - 增加向后兼容字段并保留旧字段一版周期。

## Issue P0-3：前端失败态排障助手

- 目标：
  - 分析失败/超时时提供可执行排障引导与一键复制错误上下文。
- 验收标准：

- 风险：

- 回滚方案：

## 二十、`report_analysis_events` MVP 设计（落地版）

## 1) DDL 草案

```sql
CREATE TABLE IF NOT EXISTS report_analysis_events (
  id BIGSERIAL PRIMARY KEY,
  record_id BIGINT NOT NULL REFERENCES report_records(id) ON DELETE CASCADE,
  event_id TEXT NOT NULL,                    -- SSE id / 幂等键（建议 record_id-seq）
  event_type TEXT NOT NULL,                  -- analysis.pending|running|done|failed|heartbeat
  event_payload JSONB NOT NULL DEFAULT '{}'::jsonb,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  request_id TEXT NULL
);
```

## 二十一、SSE 接口契约草案（v0.1）

## 1) Stream API

- `GET /api/reports/records/{id}/events/stream`
- 必要响应头：
  - `Content-Type: text/event-stream`
  - `Cache-Control: no-cache`
  - `Connection: keep-alive`
  - `X-Accel-Buffering: no`（防代理缓冲）

## 2) 事件定义

- `event: analysis.pending`
- `event: analysis.running`
- `event: analysis.done`
- `event: analysis.failed`
- `event: stream.snapshot_required`（`Last-Event-ID` 过旧，提示客户端回拉详情）

统一 payload 示例：

```json
{
  "record_id": 123,
  "analysis_status": "running",
  "ts": "2026-04-30T07:20:00Z",
  "message": "analysis started",
  "meta": { "attempt": 1 }
}
```

## 3) 重连语义

- 客户端断开重连时自动携带 `Last-Event-ID`。
- 服务端若能回放则补发缺失事件；不能回放则发 `stream.snapshot_required`。
- 客户端收到 `stream.snapshot_required` 后立即调用 `GET /api/reports/records/{id}` 对齐状态。

## 三十三、`routes_reports.py` SSE 端点伪实现草稿

```python
@router.get("/api/reports/records/{record_id}/events/stream")
async def stream_report_events(record_id: int, request: Request, x_reports_token: str | None = Header(default=None)):
    _require_reports_token(x_reports_token)
    last_event_id = request.headers.get("last-event-id")

    async def event_gen():
        # 1) 若 last_event_id 可回放，先补发增量事件
        for evt in load_events_since(record_id, last_event_id):
            yield sse(evt.event_id, evt.event_type, evt.event_payload)

        # 2) 持续流：监听新事件（或轮询 events 表）
        while True:
            if await request.is_disconnected():
                break
            evt = await next_event_or_timeout(record_id, timeout_seconds=15)
            if evt is None:
                yield ": keepalive\n\n"
                continue
            yield sse(evt.event_id, evt.event_type, evt.event_payload)

    return StreamingResponse(
        event_gen(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        },
    )
```

## 三十四、`reports.ts` 接入伪代码骨架（SSE + fallback）

```ts
function startAnalysisEventStream(state: ReportsState, id: number) {
  stopAnalysisEventStream();
  const url = apiUrl(state.basePath, `/api/reports/records/${id}/events/stream`);
  const es = new EventSource(url, { withCredentials: false });
  let failCount = 0;

  es.addEventListener("analysis.running", () => failCount = 0);
  es.addEventListener("analysis.done", async () => {
    await loadReportDetail(state, id, { soft: true });
    stopAnalysisEventStream();
  });
  es.addEventListener("analysis.failed", async () => {
    await loadReportDetail(state, id, { soft: true });
    stopAnalysisEventStream();
  });
  es.onerror = () => {
    failCount += 1;
    if (failCount >= 3) {
      stopAnalysisEventStream();
      startAnalysisPoller(state, id); // 退回现有轮询
    }
  };
}
```

## 六十、PR-2 `reports.ts` transition 伪代码（可直接改造）

```ts
type AnalysisUiState =
  | "idle"
  | "sse_connecting"
  | "sse_streaming"
  | "polling_fallback"
  | "terminal_done"
  | "terminal_failed";

function transition(state: AnalysisUiState, event: string): AnalysisUiState {
  if (state === "idle" && event === "OPEN_DETAIL") return "sse_connecting";
  if (state === "sse_connecting" && event === "SSE_OPEN") return "sse_streaming";
  if (state === "sse_connecting" && event === "SSE_FAIL_MAX") return "polling_fallback";
  if (state === "sse_streaming" && event === "EVT_DONE") return "terminal_done";
  if (state === "sse_streaming" && event === "EVT_FAILED") return "terminal_failed";
  if (state === "sse_streaming" && event === "HEARTBEAT_TIMEOUT") return "polling_fallback";
  if (state === "polling_fallback" && event === "POLL_DONE") return "terminal_done";
  if (state === "polling_fallback" && event === "POLL_FAILED") return "terminal_failed";
  if ((state === "terminal_done" || state === "terminal_failed") && event === "RETRY") return "sse_connecting";
  return state;
}
```

  - 服务端每条事件写入单调递增 `event_id`，并在 SSE 帧中同时输出 `id:` 与 `data.event_id`。
  - 客户端重连后利用 `Last-Event-ID` 做断点续传，先补漏再追实时。
  - 增加心跳注释帧（例如 15s 一次）降低代理层空闲断连概率。
  - 通过 `retry:` 动态控制重连间隔（高负载时适当拉长）。
- 该方案的收益：
  - 防止“分析中事件丢失但状态仍前进”的可观测断层。
  - 前端刷新或网络抖动后无需整页回拉，降低后端压力。
  - 为后续“事件重放调试”与“审计追踪”打基础。

### 155.2 前端优化：Lit 大列表与可视区切换防抖

- 对 `@lit-labs/virtualizer` 的治理建议从“仅使用”升级到“运行态防护”：
  - 列表不可见（`display:none` 或折叠容器）时暂停或降频测量，恢复可见后再重建可视区。
  - 在数据源大幅替换（大集合 -> 小集合）时主动清理旧测量缓存，避免滚动高度污染。
  - 对快速拖动滚动条场景增加“跳跃冻结 + 稳定后恢复”策略，减少抖动回流。
  - 将滚动容器模式显式化（self/ancestor/managed），避免宿主布局变化引发误判。
- 与当前周报页面的直接对应：
  - `reports-tab` 的表格/卡片切换与筛选联动后，滚动状态应统一复位策略；
  - 数据切周时先做“轻骨架 + 分段渲染”再绘图，减少首屏阻塞感。

### 155.3 周报内容逻辑：增加 DQ Gate（数据质量闸门）最低基线

- 在 `analyzer` 入参与聚合后新增可配置质检项（失败即标记 degraded 并输出告警）：
  - 完整性：关键字段空值率（`transDate/customer/totalAmount/statusName`）；
  - 唯一性：订单号重复率；
  - 范围性：金额异常阈值（负值、极端离群）；
  - 一致性：周级汇总与明细回算差值；
  - 分布稳定性：订单量/金额较上周偏移阈值（先用规则阈值，后续再引入统计检测）。
- 先从“规则阈值报警”起步，后续再升级到自动化异常检测模型，可避免一次性过度建设。
