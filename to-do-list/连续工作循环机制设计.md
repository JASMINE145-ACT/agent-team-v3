# 连续工作循环机制设计

## 问题描述

### 现象

`match_quotation_batch` 工具存在批次上限（默认 20 条），超出时仅处理前 N 条，返回文本提示用户剩余项需要再次调用。

```
⚠️ 本次仅处理了前 20 个产品（单次上限 20 条）。
还有 3 个产品未查询，请再次调用 match_quotation_batch...
```

**问题**：工具只做了"执行 + 文本提醒"，没有结构化的指令让上层 agent 自动继续。用户必须手动说"继续"才能触发第二次调用。

### 根本原因

1. **工具层面**：提醒只是自然语言文本，不是结构化控制信号
2. **Agent 层面**：没有"检测未完成项并自动继续"的循环逻辑，也没有被赋予这个职责
3. **System Prompt 层面**：行为规范缺失，agent 不知道"还有剩余"时应该自动继续

### 理想行为（对标 Claude Code）

Agent 自动完成全部查询，用户只看到最终汇总结果，无需手动干预：

```
用户：查询这23个产品
Agent：
  [自动调用 match_quotation_batch，处理前20个]
  [检测到还有3个未完成，自动再次调用]
  [汇总全部23个结果]
用户：看到完整的23个产品报价
```

## 已有机制：`[已渲染到前端]` Marker

### 来源

Trellis 中已有完整记载，见以下文档：

| 文档 | 位置 |
|------|------|
| `spec/backend/quality-guidelines.md` | § Scenario: `[已渲染到前端]` hard-stop |
| `spec/frontend/component-guidelines.md` | Card state、Marker 定义 |
| `spec/backend/skills-system.md` | "When Observation Starts with `[已渲染到前端]`" |

### 核心设计

```
工具执行 → 通过 SSE 推送卡片到前端 → 返回 "[已渲染到前端]" marker → Agent hard-stop
```

### 契约

```python
_RENDERED_MARKER_PREFIX = "[已渲染到前端]"

规则：
- 如果返回内容以 marker 开头 → 跳过下一个 LLM generation step
- 强制用 marker 文本 final answer 当前轮次
- 防止 LLM 在卡片推送后再生成额外叙述导致 query/card 不同步
```

### 前端组件约定

```typescript
// Card state
ToolRenderItem { id, runId, seq, ts, sessionKey?, payload }

// Marker
RENDERED_MARKER = "[已渲染到前端]"
```

### 局限性

这个机制**只处理了"卡片已推送，不用再叙述"**，但**没有处理"卡片没推完，剩余怎么办"**。

Marker 告诉 agent "别说了"，但没告诉 agent "还有 3 个没处理完"。

```
当前机制能阻止：LLM 在卡片推送后继续啰嗦 ✅
当前机制不能：检测到还有剩余时自动继续 ✅
```

---

## 问题分析

Agent **确实读懂了**"还有3个没查完"，但**没有行动指令**。

```
Agent 理解：⚠️ 还有3个没查完
Agent 能力：知道可以再调用 match_quotation_batch
Agent 行为：❌ 停下来等用户说"继续"
```

这不是能力问题，是**行为规范缺失**。Agent 不知道"还有剩余"时应该自动继续。

---

## 解决方案（三条路）

### 方案 1：System Prompt 规则（最快）

在 system prompt 中加入一条行为规范。

**改动点**：找到 system prompt 配置文件，加入指令

```
当工具返回提示还有未完成的项目时（如"还有X个产品未查询"），
你应该自动继续调用工具完成剩余部分，不需要等待用户确认。
```

| 优点 | 缺点 |
|------|------|
| 改一处，所有工具都生效 | prompt 会逐渐臃肿 |
| 成本最低、见效最快 | 长期难以维护 |
| 无需改代码逻辑 | 依赖 LLM 理解并遵守 |

**适用场景**：快速修复、短期方案

---

### 方案 2：工具结构化返回值 + Agent 自动循环（推荐）

两步走：

**Step 1**：工具返回标准化信号
```python
response_payload = {
    "batch_mode": True,
    "has_more": len(remaining_keywords) > 0,      # 是否有剩余
    "remaining_count": len(remaining_keywords),   # 剩余数量
    "remaining_keywords": remaining_keywords,     # 剩余关键词列表
    # ... 现有字段
}
```

**Step 2**：Agent 主循环检测并自动继续
```python
def _should_continue(result: dict) -> bool:
    return result.get("has_more", False) and result.get("remaining_count", 0) > 0

# 在工具执行后
if _should_continue(result):
    remaining = result.get("remaining_keywords", [])
    for kw in remaining:
        tool_call_queue.append(make_match_quotation_call(kw))
```

| 优点 | 缺点 |
|------|------|
| 标准化，通用性强 | 需改工具 + agent 两处 |
| Agent 行为可控、明确 | 需确保所有批量工具都返回 has_more |
| 长期可维护 | |

**适用场景**：中期方案，适合所有批量工具

---

### 方案 3：工作流层面 —— 任务分片 + 自动调度（长期架构）

引入任务队列抽象，每次工具调用视为一个"分片"，agent 自动调度直到全部完成。

| 优点 | 缺点 |
|------|------|
| 架构最完整 | 改动极大 |
| 适合复杂多步骤场景 | 引入新概念，学习成本高 |
| 可视化任务进度 | 需要新增调度模块 |

**适用场景**：长期架构升级

---

## 三条路对比

| | 方案1 System Prompt | 方案2 结构化+Agent循环 | 方案3 工作流分片 |
|--|-------------------|---------------------|-----------------|
| 改动范围 | prompt 配置 | 工具 + agent | 架构层 |
| 改造成本 | 低 | 中 | 高 |
| 通用性 | 中（依赖 LLM 理解） | 高（标准化信号） | 高 |
| 维护性 | 差（prompt 臃肿） | 好 | 最好 |
| 适用阶段 | 快速修复 | 中期标准 | 长期架构 |

---

## 与现有 `[已渲染到前端]` 机制的关系

```
现有机制：[已渲染到前端] → Agent hard-stop（不再说话）
新增机制：has_more=true → Agent 继续执行（还有活儿）

两者互补：
- [已渲染到前端] 解决"不用再说"的问题
- has_more 解决"还需继续"的问题
```

---

## 推荐路径

**近期**：方案 1（system prompt）+ 方案 2 的工具部分（`has_more` 返回值）同时进行

**中期**：方案 2 的 agent 部分（自动循环检测）配合方案 1 的 prompt 规则

**远期**：方案 3 架构升级（如果业务复杂度持续增长）
